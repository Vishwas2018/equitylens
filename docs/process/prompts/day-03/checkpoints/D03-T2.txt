### Audit module
- apps/web/server/audit/log.ts
  - hashId(id): sha256(id + PEPPER) — no raw IDs in audit trail
  - hashTenant(orgId): alias for hashId
  - canonicalJson(val): recursive key-sorted JSON (deterministic)
  - appendAuditEntry(entry): anonymises payload → gets prevHash from DB →
    computedHash = sha256(prevHash || canonicalJson(anonymised)) → inserts
  - generateInviteToken(): 32-byte random hex + sha256 hash

- apps/web/server/audit/verify.ts
  - verifyAuditChain(orgId?): fetches rows ordered by occurred_at, recomputes
    each expected_computed_hash from stored prev_hash + canonicalJson(metadata),
    returns { totalRows, verified, firstBrokenId }

## Deviations Logged

DEV-0013: inviteMember uses a one-time token rather than Supabase Auth invite flow.
          The magic-link-disabled policy applies to sign-in; invite token is a
          one-time membership grant (distinct flow). Interpretation: allowed.
          Severity: low, accepted.

DEV-0014: appendAuditEntry fetches prev_hash in a separate query then inserts.
          Under concurrent writes, two entries could share the same prev_hash,
          branching the chain. Risk is low for MVP (single-region, low volume);
          a SECURITY DEFINER atomic function is the proper fix for production.
          Severity: low, tracked for Day 8.
          -> technical-debt.md updated.

## Hash Chain Spec Compliance

  computed_hash = sha256(prev_hash || canonicalJson(anonymisedPayload))
  First entry: prev_hash = '0' * 64
  Anonymised payload keys: actor_user_id_hash, tenant_id_hash, action,
    resource_type, resource_id, occurred_at, ip_hash, user_agent_hash
  Hashing function: sha256(id || AUDIT_HASH_PEPPER)
  Verification: re-canonicalise stored metadata JSONB, recompute expected hash,
    compare to stored computed_hash
